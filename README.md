# Pitch and Crease

Daily tactical analysis publication for football and cricket. Articles are generated by Claude (using your Max subscription via Claude Code, locally), reviewed in an admin dashboard, and published to a public Next.js site hosted on Railway. Twitter posting wires in once API keys are configured.

## What's built

- **Public site** — homepage, football & cricket landing pages, article detail pages with full Open Graph + Twitter card SEO metadata.
- **Admin dashboard** at `/admin` — password-protected.
- **Article editor** at `/admin/article/<id>` — edit title, dek, slug, body HTML, status, and schedule date/time.
- **Generation pipeline** — local script that calls the Claude Code CLI on your Max subscription (no API credits). Five built-in slots plus a custom prompt mode. Drafts land in the dashboard for review.
- **Scheduled publishing** — articles with status `SCHEDULED` and a past `scheduledFor` time get auto-flipped to `PUBLISHED` by `npm run publish:scheduled`.
- **Two seeded articles** — Atlético 1-2 Barcelona (football) and "Bazball three years in" (cricket).
- **Postgres + Prisma** — single Article table covers both sports.
- **Auth** — single admin password + HMAC-signed session cookie. Sufficient for one editor.

## Coming next (tomorrow)

- Facebook page integration (gated on Meta app review — long timer, start in parallel).
- Cron job on Railway running `npm run publish:scheduled` every 5 minutes.

## Local setup

You need Node 20+ and Postgres 14+ locally. (If you don't have Postgres locally, skip ahead to "Deploy to Railway" — Railway provides Postgres and you can develop against it remotely.)

```bash
# 1. Install
npm install

# 2. Copy env, fill in DATABASE_URL + ADMIN_PASSWORD + SESSION_SECRET
cp .env.example .env
# Generate a random session secret:
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"

# 3. Push the schema to your database (creates tables)
npm run db:push

# 4. Seed the Atleti article so the site isn't empty
npm run db:seed

# 5. Run
npm run dev
```

Open `http://localhost:3000` to see the public site. Open `http://localhost:3000/admin` to log in (uses `ADMIN_PASSWORD` from `.env`).

## Deploy to Railway

You already have a Railway project (`cc1555df-…`) with Postgres added. Here's the rest:

### 1. Create the GitHub repository

```bash
cd pitch-and-crease
git init
git add .
git commit -m "Initial scaffold: Pitch and Crease"
git branch -M main
# Create the repo on GitHub at https://github.com/new (name it `pitch-and-crease`)
git remote add origin https://github.com/Rohit-Patekar1/pitch-and-crease.git
git push -u origin main
```

### 2. Connect the repo to Railway

In the Railway project:

1. Click **`+ New`** → **`GitHub Repo`** → authorize Railway to read your repos if prompted.
2. Pick **`Rohit-Patekar1/pitch-and-crease`**.
3. Railway will detect Next.js and start a build. The build will FAIL the first time because env vars aren't set — that's expected.

### 3. Set environment variables

Go to the new service → **Variables** tab → add these (each key on its own row):

| Key                    | Value                                            |
|------------------------|--------------------------------------------------|
| `DATABASE_URL`         | Click "Add Reference" → pick the Postgres → `DATABASE_URL`. Auto-fills. |
| `ADMIN_PASSWORD`       | A long random string. Save it somewhere — this is your login. |
| `SESSION_SECRET`       | Run `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"` and paste. |
| `SITE_NAME`            | `Pitch and Crease`                               |
| `SITE_URL`             | Will be `https://<service-name>.up.railway.app` once Railway gives you the public domain. Update after first deploy. |

Twitter env vars come later (`TWITTER_API_KEY`, `TWITTER_API_SECRET`, `TWITTER_ACCESS_TOKEN`, `TWITTER_ACCESS_SECRET`). Leave them blank for now.

### 4. Generate the public domain

In the service → **Settings** → **Networking** → **Generate Domain**. You'll get something like `pitch-and-crease-production.up.railway.app`.

### 5. Trigger a deploy

Railway redeploys automatically when env vars change. If not, click **`Deploy`** on the latest commit.

The build runs:

```
prisma generate          # creates the typed client
prisma db push           # creates tables on the Postgres
next build               # builds the site
```

### 6. Seed the database (one-time)

After the first successful deploy, the database has empty tables. To load the Atleti article:

```bash
# Locally, with your .env pointed at the Railway DATABASE_URL temporarily:
DATABASE_URL="<paste Railway Postgres URL>" npm run db:seed
```

Or — easier — open the Railway service shell (Settings → "One-off command") and run:
```
npm run db:seed
```

Visit your Railway URL. You should see "Pitch and Crease" with the Atleti article on the homepage.

## How articles get into the system

### Manual (always available)
Edit `prisma/seeds/*.html`, run `npm run db:seed`. Or insert directly via Prisma Studio (`npm run db:studio`).

### Generation via Claude Code (Max subscription, no API credits)

Prerequisite — install and log in to Claude Code on your laptop (one-time):

```bash
# Install
curl -fsSL https://claude.ai/install.sh | bash
# (or: brew install claude  on macOS)

# Authenticate against your Claude Max account (browser-based OAuth)
claude /login
```

Then, from the project folder, run any of these. Each one creates a `DRAFT` article:

```bash
# Specific slots
npm run generate -- --slot=on-this-day
npm run generate -- --slot=recent-match
npm run generate -- --slot=transfer
npm run generate -- --slot=tactics
npm run generate -- --slot=player

# Custom prompt
npm run generate -- --slot=custom --custom="Tactical analysis of De Zerbi's high-line at Marseille this season"

# All five slots in sequence (~5-10 minutes total)
npm run generate -- --daily
```

Each run takes 1-3 minutes. Drafts appear in the admin queue; click any one to review/edit, then change status to `APPROVED` or `PUBLISHED`.

### Scheduled publishing

Set `status = SCHEDULED` and a `scheduledFor` time on an article. Then run:

```bash
npm run publish:scheduled
```

Any SCHEDULED article whose time has passed gets flipped to PUBLISHED. You can run this on a 5-minute cron — Railway Cron, GitHub Actions, or `launchd` on your Mac.

## Tech notes

- **Article body storage**: each article's full HTML body (including its `<style>` block and embedded SVG) lives in a single `body` text column. The seed/generation step rewrites global selectors (`:root`, `body`, `html, body`) so they only apply within `.article-body`, preventing the article's design from bleeding into the rest of the site.
- **SEO**: every article page emits `og:`, `twitter:`, and `article:` metadata. We'll add `JSON-LD` Article schema in week 2 once we have a publish-author flow.
- **Performance**: pages are server-rendered with Prisma reads. Postgres queries are simple indexed lookups; Railway's free Postgres handles thousands of articles comfortably.
